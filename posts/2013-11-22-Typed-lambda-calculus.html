November 22, 2013 (Friday) Type theory

Typed lambda calculus

In this post, I briefly introduce typed lambda calculus in its simplest form and explain the type inference rules. The Curry Howard Isomorphism appears for the first time here although of a toy logical system, namely the propositional logic with implication (⇒) as the only boolean operation.

What are types ?

We fix a set 𝒯 of base types that can be thought of as the built-in types of the language. For this post, the set of types are all terms generated inductively as the follows:

τ≔t∣τ1→τ2

In the above definition t is some basic type, i.e an element of the set 𝒯, and τ1 and τ2 are themselves types in which case τ1→τ2 is the type of functions from τ1 to τ2. There are few points that we want to clarify right away.

  1. Consider the set of propositional logic formulas where the base set of propositions is 𝒯 and the only logical connective is the logical implies ⇒. We can see that there is a one-to-one correspondence with types of our language: interpret the functional type symbol → as the implication operator ⇒. We will see in this post that this connection is not just skin deep.

  2. The types in our language are defined independent of the terms in our lambda calculus. When we want more powerful types the will break. We will have to define the values and types together.

What is a well typed expression ?

Intuitively, a typed lambda calculus is a version of lambda calculus where each expression is assigned a type. The property that we want to ensure is that whenever an expression f of type σ is applied on another expression e of type τ, the type σ should be a function type from τ→τ′, for type τ′. Whatever systematic procedure we adopt to assign types to expressions, the the base case to handle therefore should be variables. There are two kinds of variables that can occur in an expression, free and bound. The bould variable gets its type from the site of binding by ensuring that the the only λ-abstraction that are allowed are of the form λx:τ.e. In such a case every free occurance of x in e is assumed to carry the type τ. For the free variables the only way we can have types is by type assumptions. This is not unexpected: in any programming language the meaning of the program dependes on what the free variables are. Something similar is happening here.

We will use the notation e:τ to say that e has type τ. A type assumption Γ is a set of assertions of the kind x:τ where x is a variable. Ofcourse we disallow the same variable to have two types in an assumtion.
