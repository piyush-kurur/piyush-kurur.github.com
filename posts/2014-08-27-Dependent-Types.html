Dependent types

In the last post, we looked at simply typed lambda calculus where each term was associated with a type. As such the types of the language and terms were completely separated ‚Äî we could define the types of the language independent of terms. A dependently typed language is one where the types can depend on values from a given type. For example consider the natural numbers ‚Ñï. In a language like Haskell, we can define them inductively. In the last post, we saw that the type inference rules of simply typed lambda calculus gave us the rules of natural deduction for (a fragment of) propositional logic via the Curry-Howard isomorphism. To extend this to predicate logic, we would need a dependently typed lambda calculus as we explain now. Type inference rules.

So not just the free variables in e but all free variables that occur in A should appear in the type assumption. Thus our strategy of defining types independent of terms like in the previous post will not work; we need to define them hand in hand.

The type assumptions themselves should now be an ordered sequence of assumptions of the form x:A with the added restriction that any assumption x:A in the sequence, all the variables that appear free in the type A should themselves be defined previous assumptions in the sequence. Where are the dependent types?

The dependently typed language we introduced here is mostly useless in the absence of any interesting types and type families. One strategy would be to introduce each type and type family by hand giving its, formation, introduction and elimination rules. At the very least, we fix a base set of types ùíØ and add the formation rule ttypet‚ààùíØ. This will give us all the types described in the previous post (once we identify Œ†x:AB with A‚ÜíB whenever x is not free in B). To make dependently typed systems more useful actual systems supports construction of user defined types and type families. What these constructions are and what should be the restrictions on them has to be worked out. We defer this topic for the future.
